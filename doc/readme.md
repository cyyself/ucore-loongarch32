## 项目概述

ucore是用于清华大学计算机系本科操作系统课程的OS教学实验内容，起源于MIT CSAIL PDOS课题组开发的xv6&jos、哈佛大学开发的OS161教学操作系统、以及Linux-2.4内核。其中，清华大学基于ucore的OS教学实验是基于X86架构。

LoongArch是龙芯中科推出的指令系统，包括基础架构部分和向量指令、虚拟化、二进制翻译等扩展部分。LoongArch在架构上进行自主重新设计，具有较好的自主性、先进性与兼容性。

我们将ucore移植到了用于教学目的的LoongArch32指令平台，并成功运行了所有的用户程序。

本项目目标如下所示：
1.	阅读理解目标系统和LoongArch32架构特性，实现LoongArch架构支持。
2.	搭建LoongArch交叉编译环境，编译目标系统，并在QEMU模拟器上完成功能调试，形成移植和测试文档。
3.	移植相应的课程实验。


## 实现思路

1. 选择一个ucore版本（这里基于[z4yx/ucore-thumips](https://github.com/z4yx/ucore-thumips)为基础进行移植）
2. 熟悉工具链，跑起一个LoongArch32的裸机程序
3. 阅读LoongArch32文档，对比与MIPS的差异，进行移植，并调试。

## 实现过程

### 裸机程序的实现

在实现裸机前，首先阅读了LoongArch32文档，发现不同于MIPS在固定的特权级对虚拟内存地址空间有固定的片段（如kseg0、kseg1等），LoongArch32采用了直接映射配置窗口与软件控制TLB相结合的方式对地址进行配置，而LoongArch32提供了2个映射配置窗口，每个窗口配置虚拟地址高3位到物理地址高3位的映射，且该机制优先于虚拟内存管理的TLB机制。

此时引申出了一个问题，qemu通过-kernel载入ELF时，如何将ELF中的虚拟地址对应到后续才要配置DMW的物理地址并启动？

为了解决这一问题，通过readelf与objdump观察了Linux的启动代码，并多次测试，结果发现QEMU在开启地址翻译前，对于所有的内存地址访问（包括取指与Load/Store）都会去除最高3位，因此这就给了操作系统启动时对地址翻译窗口进行设置的机会，在操作系统链接时也就可以直接使用虚拟地址而不必考虑相对地址问题。

然后，为了实现裸机程序的IO，需要找到QEMU中对应的外设地址，最后发现，QEMU内实现了一个UART8250串口，基地址位于物理地址0x1fe001e0。

最后，编写了一个简单的asm脚本初始化各CSR以及栈指针进入C程序，加上一个简单的C程序往串口发送数据，与一个简单的ld脚本将他们链接在一起并指定入口物理地址，完成了一个裸机程序Hello World实现。

### 开发环境

经过与龙芯沟通，我们拿到了LoongArch32平台的相关工具链，我们我们将环境统一打包为一个docker容器，该容器已经将qemu，编译工具链和gdb配置好，直接make就能使用，以便后续用于教学用途时，学生们能够直接使用该Docker容器完成实验，而不必进行复杂的环境配置以及问题解决。

同时，尽管我们打包的Docker容器基于x86架构，我们同样考虑了学生们可能会使用非x86架构进行开发的问题，在实验指导书中讲述了binfmt与qemu-user的使用，使我们打包的工具链能够运行在这些架构上，并经过在树莓派上实际运行的验证，确认转译效率足以完成操作系统实验。

### ucore的内核入口点移植

对于QEMU虚拟机而言，我们可以直接使用-kernel来指定我们需要加载的内核的ELF文件，从而直接完成了内核复制到内存的载入过程，并直接从ELF的入口点开始启动。ucore内核加载到内存后，就会跳转到ucore操作系统在内存中的入口地址(kern/init/entry.S中start的地址)，这样ucore就接管了整个控制权。

`kern/init/entry.S`中主要需要移植的内容如下:

1. 配置DMW，使得操作系统拥有可用的地址空间

​	龙芯结构32位精简版的MMU支持两种虚实地址翻译模型:直接地址翻译模式和映射地址翻译模式，当CSR.cRMD的DA=1且PG=0时，处理器核处于直接地址翻译模型，这种模式下物理地址默认直接等于虚拟地址的[PALEN-1:0]位，当CRMD的DA=0且PG=1时，处理器核的MMU处于映射地址翻译模式，该模式下可以通过直接映射配置窗口机制完成虚实地址的直接映射。直接映射配置窗口设置有两个，系统软件可以通过配置DMW0~DMW1寄存器来分别设置两个直接映射配置窗口。每一个直接映射配置窗口可以配置一个512M（即32位中去除最高3位的地址空间）固定大小的虚拟地址空间，当虚地址命中(虚地址的最高三位与配置窗口寄存器中的高三位相等)某个有效的直接映射配置窗口时，其物理地址直接等于需地址的[28:0]位凭借上该映射窗口所配置的物理地址高位。

​	我们最终的ucore启动阶段配置了2个DMW，结合基于软件控制TLB的虚拟地址翻译后，虚拟地址空间如下：

```
+-----------------+  ->0xc0000000
|  Kernel Cached  |  DMW0：Cached，Mapped to 0x00000000-0x1fffffff
+-----------------+  ->0xa0000000
| Kernel Uncached |  DMW1：Uncached，Mapped to 0x00000000-0x1fffffff
+-----------------+  ->0x80000000
|   Userprogram   |  Software controlled TLB translate
+-----------------+  ->0x00000000
```

2. 设置CSR.PRMD寄存器。

   在LoongArch32架构中，CSR.PRMD寄存器控制的是当前运行模式信息。在ucore内核启动中，设置CRMD即存取的PLV域为0。IE域为0，DA域为0，PG为1:PLV表示当前特权等级，其合法的取自范围为0和3，0表示最高特权等级（即我们ucore的运行特权级），3表示最低特权等级。IE是全局中断使能，当触发例外时，硬件将该域的值置为0，以确保陷入后屏蔽中断。DA=0，PG=1时，处理器核的MMU处于映射地址翻译模式。当DA=1，PG=0时，处理器核的MMU处于直接地址翻译模式。

### TLB清空

由于LoongArch32架构不保证处理器复位时完成TLB的清空，为了防止存在脏TLB表项干扰后续的系统运行，需要在使用虚拟内存之前完成TLB的清空，这里直接使用了LoongArch32的INVTLB指令完成TLB的清空。并在`kern_init`中初始化内存管理前进行调用。

具体代码位于`kern/mm/la32_tlb.c`。

### 常规例外实现

中断与例外是ucore操作系统实验1的要求。LoongArch32架构的中断与MIPS类似，在CSR（MIPS中为CP0）提供了一系列中断使能与记录寄存器，并对于所有的例外（除TLB重填外，包括中断）提供了一个例外入口地址配置寄存器，以及TLB重填的例外入口地址配置寄存器。

配置例外入口的代码位于`kern/init/init.c`，也是`kern_init`调用的第一个函数，该函数将例外入口地址写入`CSR.EBASE`寄存器中。而地址来源是一个`extern`的符号`exception_handler`，该符号对应的汇编代码位于`kern/trap/exception.S`。

在`exception_handler`中，完成了所有通用寄存器与CSR中必要的异常信息保存到trapframe，并修改栈指针寄存器（即`sp`）指向内核栈，然后进入位于`kern/trap/trap.c`中的`loongarch_trap`函数开始处理。

这里做的主要移植包括删去了MIPS上需要保存的用于乘法的HILO寄存器，并增加了不再用于保留用途的通用寄存器，以及重新定义了需要保存的CSR以及修改了后续使用这些CSR的地方，并重新定义了异常类型对应异常号的宏。

### 中断实现

在中断管理上，LoongArch32架构将各个中断使能位放置于`CSR.ECFG`寄存器中，并将全局中断开关置于`CSR.CRMD`中，并在进行异常处理（包括中断）时会自动将全局中断关闭。

在QEMU中，来自CSR定时器的时钟中断号为11，来自UART串口的中断号为2，这一部分代码定义在`kern/include/loongarch.h`中，并在初始化对应设备的时候对`CSR.ECFG`寄存器相应的二进制位设置1。

对于时钟中断，LoongArch32架构自带的时钟是在CSR.TCFG(0x44)中进行配置，这里要配置寄存器的使能位En为1，设置定时器循环模式控制为Periodic为1，即定时器在倒计时自减至0时，在置起定时中断信号的同时，还会自动定时器重新装载成TimeVal域中配置的初始值，然后再下一个时钟周期继续自减。设置InitVal域，指定定时器倒计时自减计数的初始值。最后开启定时器中断，即修改控制寄存器ECTL定时器中断源的使能位的第13个中断号，之后就能不断观测到时钟中断的产生进入例外处理地址了。具体代码位于`kern/driver/clock.c`。

对于串口中断， 由于UART8250设备相同，因此在我们移植处理好了它的基地址和中断号之后即可直接工作。

### 虚拟内存管理实现

与MIPS架构类似，LoongArch32的内存管理是使用软件直接管理TLB的方法，当处理器的TLB无法命中时，就产生一个TLB重填的异常，跳转到TLB异常处理地址，由软件完成TLB的重填。

解决该问题的第一步就是解决TLB异常地址入口问题，LoongArch32架构中，TLB重填异常提供了一个专用的例外入口，并在TLB重填异常产生时，会修改`CSR.CRMD`寄存器使处理器完全运行在物理地址模式，关闭包括DMW在内的所有虚拟内存管理特性，这就导致我们对于TLB重填异常不能直接按照传统异常处理。

针对这一问题，移植过程中实现了一个虚拟内存管理的跳转机制，涉及到`tlbrefill_handler`和`tlbrefill_redirector`两个函数，位于`kern/trap/exception.S`中。

首先将`tlbrefill_redirector`对应的物理地址在内核启动阶段保存在一个硬件寄存器中（最终选择了`CSR.KS3`），以便能够方便地访问该地址。并将TLB重填的例外入口地址（`CSR.RFBASE`）设置为`tlbrefill_handler`的物理地址，以便TLB重填例外产生时能够跳转到该地址。

之后，编写`tlbrefill_handler`与`tlbrefill_redirector`两个函数。在`tlbrefill_handler`中需完成`t0`寄存器的保存（以便函数自身使用）、`CSR.PRMD`的保存与清零、`CSR.EPC`的保存并指向`tlbrefill_redirector`，然后使用ertn指令，使处理器切换回虚拟地址模式，启动DMW虚拟内存机制，同时留下例外现场的`CSR.PRMD`和`CSR.EPC`和`t0`寄存器的值。在`tlbrefill_redirector`中，从保存位置复原`CSR.PRMD`、`CSR.EPC`、`t0`寄存器的值，然后跳转`common_exception`，从而使TLB例外经过跳转能够像传统例外一样工作在基于DMW的虚拟内存地址中，以便进入ucore内核中C语言编写的例外处理函数完成TLB的重填。

在TLB的格式上，MIPS与LoongArch32格式几乎完全一致，同样采用了虚双页表的形式，仅在页面大小表示方法存在不同，因此这方面的移植就只需要将对应CSR寄存器的位域定义进行修改即可。

### 系统调用实现

由于LoongArch32中，v0、a0与v1、a1是相同寄存器，导致原先MIPS中通过v0传递系统调用号，用a0-a3传递系统调用参数的方式需要进行移植。为此，我们将系统调用号修改为了a7，参数的传递继续从a0开始，就解决了系统调用的移植。

## 开发过程中遇到问题

我们在开发过程中遇到几个比较值得注意的问题，供其他为LoongArch32的开发操作系统的人员参考：

### 1. CSRWR指令会替换rd寄存器中的值

这一点与MIPS的mtc0、mfc0不同，这一点导致我们在编写异常处理入口的汇编代码时遇到了一些问题，好在通过qemu+gdb发现了这一现象，并仔细阅读文档后发现确实在文档中介绍了这一特性。

### 2. 存在重名的寄存器

在LoongArch32中，v0、a0与v1、a1是相同的寄存器，但用了不同的名字，而MIPS中不是这样，这就导致ucore之前定义的系统调用传递参数与系统调用号的方式与这一设计存在冲突，在定位该问题时也花了较长的时间。
